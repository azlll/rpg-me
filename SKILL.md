# 人生游戏角色卡

## 这个 skill 做什么

根据用户当下的状态、性格和隐藏愿望，生成一组 RPG 角色卡，最终产出一个 HTML 结果页给用户。

重要定位：**HTML 是「结果展示页」，不是「生成器」**。所有内容（角色设定、属性、立绘）都由你（agent）在生成阶段确定后写入 HTML，网页本身不做任何 AI 生成。页面只负责漂亮地展示结果、并允许用户把 4 张卡片逐张 PNG 下载。因此：

- HTML 里不要出现输入框、"生成"按钮或任何暗示网页能再生成卡片的元素。
- 如果用户想换一张卡，是回到对话里让你重新生成，而不是在网页上操作。

## 工作流程

1. 判断输入模式（快速 / 深度），收集用户信息。
2. 设计角色：名称、称号、职业、等级、种族感、画风、专属武器、6 项人生属性、RPG 基础数值、主动/被动技能、当前任务、风味文案。
3. 生成角色立绘（见「生图方案」，务必得到一张真实可显示的图）。
4. 把本次角色数据写入历史记录：优先运行 `scripts/history_records.py`，输出到 `output/history/YYYYMMDD-HHMMSS-<角色名>/index.html`，并刷新 `output/history/index.json` 与所有历史页左侧列表。
5. 给用户一段「角色解析」文字 + 本次 HTML 结果页链接 + 历史页链接。

## 输入模式

检测到生成请求时，主动提供两种方式：

- **快速模式**：用户用一句话描述自己，例如「最近刚辞职，想 gap 一年，喜欢海边和咖啡」。
- **深度模式**：依次问 5 个问题，收集更丰富信息后再生成。

### 深度模式问题库（选 5 个）

1. 用三个词形容你现在的状态。
2. 最近最让你上头的一件事是什么？
3. 如果可以拥有一个超能力，你想要什么？
4. 你现在的「主线任务」是什么？
5. 你觉得自己更像哪种生物、自然元素或天气？
6. 你最常在什么场景下感到放松？
7. 说一个你最近的小成就或糗事。
8. 如果人生是一款游戏，你希望它是什么类型？

## 角色设计规则

整体世界观固定为**明亮童话 JRPG 冒险世界**：像勇者小队刚出新手村的轻松奇幻冒险，色彩明亮、怪物友好、城镇温暖、有任务感。不要直接写 Dragon Quest，也不要把世界观切到赛博朋克、克系、现代都市等方向。用户的现代生活元素要被翻译成奇幻 RPG 里的职业、道具、武器、护符、伙伴或小装饰。

背景地点可以变化，但都应属于同一个冒险世界，例如：新手村广场、草地小山丘、魔法道具店、冒险者酒馆、森林小路、城堡门口、海边港口、云朵神殿、地下迷宫入口、篝火营地。

人物画风可以多样。生成前如果适合，可以让用户选择；用户不选时由你根据回答推荐。可参考：

- **默认推荐**：根据用户答案选择最贴合的画风。
- **3D 手办风**：玩具感、圆润、适合可爱传播。
- **2D JRPG 风**：经典角色卡、最稳。
- **手绘童话风**：温柔、生活感、适合疗愈主题。
- **火柴人勇者风**：极简、自嘲、梗图感。
- **拼豆风**：颗粒像素玩具感，适合轻松分享。
- **方块体素风**：方块世界感，不直接写 Minecraft。
- **像素风**：复古 RPG、升级感强。
- **黏土小人风**：温暖、家庭感、治愈。
- **纸艺剪贴风**：手账感、童话书感。

默认推荐规则：疲惫/想下班/自嘲时可推荐火柴人勇者风、像素风或 Q 版；想变强/赚钱/创造时可推荐 2D JRPG 风或 3D 手办风；温柔/家庭/疗愈主题可推荐手绘童话风、黏土小人风或纸艺剪贴风；技术/搭建/系统感强时可推荐方块体素风或像素风。

### 专属武器

每个角色必须有一件**专属武器**，但武器不是固定清单，也不要按职业模板硬套。先从用户回答中提取 1 个最有辨识度的生活物件、技能工具、兴趣物、食物、职业工具、近期梗、主线任务或愿望，把它 RPG 化为武器、法器、盾牌、伙伴、召唤物、护符或魔导装置。

武器来源优先级：

1. 用户明确喜欢或常做的事，例如炒菜可以变成锅铲、炒锅、调味瓶或围裙盾。
2. 用户当前工作或技能，例如程序员可以变成 AI 助理、代码精灵、调试镜、提交印章或自动补全魔导书。
3. 用户最近上头的梗、食物或物件，例如折耳根可以变成怪味护符、绿色灵感草或草本法杖。
4. 用户当前主线任务，例如备孕可以抽象成晨光罗盘、温柔小巢盾、作息沙漏；不要画成医疗或育儿场景。
5. 用户想要的能力，例如无限创造力可以变成点子熔炉、造物扳手、灵感魔杖。

如果用户答案里有强生活物件，优先用生活物件做武器；如果没有，再用职业工具；如果职业工具也不明显，再用主线任务或愿望生成抽象武器。最终必须说明武器来自哪条用户回答、现实原型是什么、RPG 化后是什么、在图里如何露出来。

### 属性（固定 6 项，每项 1-100）

| 属性 | 含义 | 提示 |
|------|------|------|
| 体力值 | 精力与行动力 | 忙碌疲惫则低，元气满满则高 |
| 社交力 | 与人连接的能量 | 社牛高，社恐低，≠ 好坏 |
| 创造力 | 想象力与输出欲 | 爱表达、脑洞大则高 |
| 摸鱼值 | 松弛与反卷能力 | 会休息、敢摆烂则高 |
| emo 抗性 | 情绪稳定与自愈力 | 受挫则低，内心强大则高 |
| 运气值 | 近期运势与意外感 | 可带一点随机趣味 |

避免全属性都高，保留 1-2 个明显短板，角色更真实、更有梗。

### 卡牌固定结构

输出为 4 张 9:16 轮播卡，每张之间在 HTML 中纵向留出间距，方便用户手动截图或直接下载。

1. 封面卡：1:1 半身英雄立绘、角色名、职业、等级、tagline、6 项人生属性。立绘必须露出专属武器。属性只展示数值，数值前使用 SVG icon，不使用柱状图。
2. 角色解析卡：解释为什么这样设定，让短板也变成有梗、有共情的可分享文案。
3. RPG 面板卡：名称、等级、称号、力量、敏捷、智力、HP、MP、EXP、种族、特性、天赋。每个小数值前有 SVG icon。
4. 技能卡：主动技能、被动技能、当前任务（1-2 条）。

补充数值：力量、敏捷、智力、HP、MP、EXP 要和用户状态相符，避免全高；种族、特性、天赋要短、有记忆点。

## 生图方案（务必产出真实图片）

用户的 agent 环境各不相同，不能假设一定有生图能力。按以下优先级取图，**保证 HTML 里最终是一张能显示的图**。详细说明见 [references/image-generation.md](references/image-generation.md)。

1. **优先**：如果当前环境有可用的生图能力（如内置文生图工具、`autoglm-generate-image` 等 skill），直接调用，拿到图片文件或 URL。
2. **优先 API：通义万相 / 阿里云百炼**：若环境没有生图能力，运行 `scripts/generate_portrait.py`。脚本会先检测 `DASHSCOPE_API_KEY` + `DASHSCOPE_WORKSPACE_ID`（环境变量或项目根目录 `local-image-api.md`），检测到后使用通义万相 API。skill 里只保存文档 URL，不保存 key。
3. **通用 API 回退**：未配置通义万相时，脚本再检测 `IMAGE_API_BASE` + `IMAGE_API_KEY`，兼容 OpenAI 风格 / 智谱 / 自定义 endpoint。
4. **占位图回退**：以上都不可用时，使用 `assets/placeholder-portrait.svg` 作为占位立绘，并在给用户的文字里说明「未检测到生图能力，已用占位图，可配置 API key 后重新生成」。

无论用哪种方式，拿到图后都要**转成 base64 内嵌进 HTML**（见下），不要用相对路径引用外部图片文件，否则用户移动或分享 HTML 时会看不到图。

生图提示词结构：`[明亮童话 JRPG 冒险世界] [人物画风] [角色身份/种族感] [服装/装备] [专属武器名称与外观] [姿势] [背景地点] [氛围光] [构图硬约束]`。立绘槽是 1:1 方形区域，通义万相默认生成 `1080*1080`；提示词必须包含 `square composition, upper-body hero portrait, waist-up or chest-up framing, centered subject, face clearly visible, weapon clearly visible, enough headroom, visible torso and hands, simple fantasy background, no full-body shot, no tiny character, no text, no logo, no watermark`。人物为主体、背景简洁、无文字。若用户没给外貌信息，不要默认东亚人脸，可用剪影、动物拟人、Q 版、机械体等。

## HTML 输出

使用 [assets/card-template.html](assets/card-template.html) 作为模板。推荐不要手动直接写 HTML，而是把角色数据保存成 JSON 后运行 `scripts/history_records.py`，让脚本统一替换模板、生成独立历史目录并刷新历史索引。模板特性：

- 纯展示页，无任何生成控件。
- 左侧有「生成历史」列表，行为类似聊天记录列表；点击历史 item 会在当前 HTML 内切换显示对应角色卡，禁用 JavaScript 时也能通过链接打开对应记录页。
- 立绘以 base64 data URI 内嵌（替换 `{{PORTRAIT_DATA_URI}}`），确保任何环境都能显示。
- 输出为纵向排列的 4 张 9:16 轮播卡，适合小红书翻页展示。
- 立绘区固定为 1:1，通义万相默认生成 `1080*1080`，确保图片适配封面卡顶部区域。
- 顶部有「下载全部 4 张」按钮，用 html2canvas（已通过 CDN 引入）把 4 张卡逐张 PNG 导出；每张卡下面也有「下载这一张」按钮。
- 不使用 localStorage/sessionStorage。

需要替换的占位符见模板顶部注释，包括角色名、职业、称号、等级、人生属性值、RPG 面板数值、种族、特性、天赋、技能、任务、tagline、解析文字、立绘 data URI、生成日期、编号与历史列表。

### 历史记录

每次生成都要成为一条独立记录，不要覆盖上一条结果。推荐流程：

1. 生成角色数据 JSON，字段名使用模板占位符名，例如 `CHAR_NAME`、`TITLE`、`STAT_HP`、`ACTIVE_NAME`、`QUEST_LIST` 等。`DATE` 和 `SERIAL` 可省略；脚本会按当前时间自动生成 `RPG-YYYYMMDD-HHMMSS`。
2. 准备立绘文件。如果没有真实立绘，使用 `assets/placeholder-portrait.svg`。
3. 运行：

```bash
python scripts/history_records.py --data card-data.json --portrait portrait.png --source-summary "用户输入摘要"
```

脚本会创建：

- `output/history/<record-id>/index.html`：本次结果页，左侧含全部历史记录。
- `output/history/<record-id>/metadata.json`：本次角色数据、编号、输入摘要和 base64 立绘。
- `output/history/index.json`：所有记录的机器可读索引，新记录排在最前。
- `output/history/index.html`：历史总入口。

当已有历史存在时，脚本会刷新所有记录页的左侧列表，因此任意一次结果页都能切换查看之前生成过的角色。

## 注意事项

- 不对用户做负面评判，短板属性用幽默或共情方式表达。
- 避免真实疾病、心理诊断、封建迷信等敏感内容。
- 文案要有「小红书感」：句子短、有画面感、适合截图。
- 未指定风格时由你自主选择，并在解析中点明为什么选这个风格。
